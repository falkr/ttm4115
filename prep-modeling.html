Deployment Diagrams

Deployment diagrams show some of the structure of a system by showing the computing nodes it consists of, how they are connected by communication channels and how software and files are assigned to them. (This assignment is also called deployment, therefore the name of the diagram.) There is no behavior involved, that means, deployment diagram show a snapshot of a system as it is, and this structure usually does not change.

Deployment diagrams consist of basically three different modeling elements:

  1. Nodes that represent hardware devices and software execution environments. In other words, nodes that can execute software.
  2. Connections that represent communication between nodes.
  3. Software artifacts that are assigned to nodes.

(Figure with these three elements).

Nodes: Devices

One type of nodes are devices, which represent hardware devices, i.e., physical things that you can take in your hands. Devices are shown as boxes with a 3D-effect, with the stereotype «device».

Stereotypes are a way to add concepts in UML that are not part of the core language. Just think of them as a pair of funny parentheses («») with a word in between.

Examples for stereotypes for devices are «pc» «raspberry pi» «phone» «tablet» «iphone» «server» «gateway»

Why is there an iPhone and a Raspberry Pi? For UML any of these are valid stereotypes if you as the designer say so. If that makes sense, depends on your project, and only you can tell. Is it an important detail that you are talking about a Raspberry Pi? If yes, fine! If not, use a more generic device type.

Nested Devices

Devices can also be nested. We can for instance show that a disk and memory is contained in a pc, or that a gateway contains a trusted platform chip.

Again, you need to decide if this is a detail relevant for your model or not. One can argue, for instance, that a trusted platform module in ah gateway is worth showing if it has significant influence on the system and the use cases it can serve.

Nodes: Execution Environments

Another type of nodes are execution environments. shown as a 3d box with <>. These are also nodes that execute software, but in contrast to devices above they consist themselves of software. An example is a web server, which acts like container for code that serve web sites or APIs (for instance called servlets.)

Examples for execution environments are:

  • «runtime environment»
  • «application server»
  • «web server»
  • «operating system»
  • «docker»
  • «Java VM»

Since execution environments are software, they need to run on a device, that means, they are contained in a device node. Again, you need to decide if showing an execution environment is relevant for your project.

I haven’t so far found a good example where an execution environment contains another execution environment. In any case, a device cannot be contained within an execution environment, since the former represents physical hardware and the latter software.
